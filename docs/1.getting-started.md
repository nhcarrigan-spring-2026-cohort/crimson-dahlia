# â–¶ï¸ Step 5 â€” Backend Setup (Flask)

If you've been following along exactly, you will need to backtrack a little bit to make your way into the backend folder.

Where you were just in `../crimson-dahlia/frontend`, you now need to be in `../crimson-dahlia/backend`.
The quickest way to do this is by going back one directory and then changing into the next, like this:

```
cd ..
cd backend
```

Just like the frontend, you will need to install the backend dependencies, but it will use different commands.

## 1. Check Python

    python --version
    pip --version

If not installed, download from: https://www.python.org/downloads/

### What is pip?
PIP is Python's package installer; it's like easy_install, and it is going to make our lives much easier when it comes to updating and changing our dependencies. 

---

## 2. Create a Virtual Environment

    python -m venv venv

Activate it:

### Windows

    venv\Scripts\activate

### Mac / Linux

    source venv/bin/activate

### Why? 
A virtual environment makes sure that the versions of all dependencies used in this app do not affect anything else that may be using the same resources globally. Even if you are working on one project, it is good practice to use a virtual environment when working with Flask.

----

## 3. Install Dependencies

    pip install -r requirements.txt

If you have any errors with imports or things you are using in the backend, try running this command first. It will update any resources you are currently using to match the code to date. Also, please update this file if you add any dependencies to the backend.
For those who are familiar with frontend: *This is the Python version of `npm install`*

---

## 4. Start Flask

    flask run

You should see:

    Running on http://127.0.0.1:5000

# ğŸŒ¿ Git Basics (Saving Your Work)

## Check what changed

```
git status
```

## Add files

```
git add . 
```
*This adds all changed and new files. If you see something unexpected, run git status to double-check.*

## Save (commit)

```
git commit -m "describe what you changed"
```

Commits should be relatively short but meaningful "bug fixes" and "new feature" don't tell us what changes you've made, but "fix(ui): navbar is now responsive when resizing the window" is clear and categorized

https://www.conventionalcommits.org/en/v1.0.0/ : refer to this file for reference of what to call your commits

## Upload to GitHub

```
git push origin <branch-name>
```

Make sure you push to your branch and not main! (replace `<branch-name>` with the name of your branch)
